# 需求调研规范 (V0.3)

## 1. 目标
本规范定义了撰写技术组件或功能需求调研报告的标准格式。其核心目标是确保调研输出能**清晰、无歧义地界定功能边界，揭示内部实现的关键矛盾与依赖**，从而为后续的架构设计、工作量评估、技术选型与风险识别提供直接、可靠的输入，避免因需求理解偏差导致的项目返工。

## 2. 文档结构要求
调研报告需严格遵循以下章节结构撰写，以确保内容的全面性、结构化和可追溯性。

### 1. 概述
-   **核心内容**：阐述该功能的**定位、核心价值**以及它旨在解决的**关键问题**。
-   **撰写要点**：避免陷入细节，应从较高维度说明此功能“是什么”和“为什么需要它”。可简要说明其目标用户和应用场景。

### 2. 接口
-   **核心内容**：**详尽列举**所有面向用户或开发者的访问接口。本规范中“接口”泛指用户与系统功能交互的契约，根据上下文可能指：**SQL语句/语法**、**编程API（函数/方法）**、**命令行工具（CLI命令）** 或**配置文件字段**。
-   **描述要求**：
    -   每个接口需独立小节（如 `2.1 创建资源`，`2.2 查询状态`）。
    -   以表格形式详细描述每个参数，包括：参数名、类型、是否必填、默认值、参数说明与取值范围。
    -   明确说明接口的返回值、输出格式或可能抛出的异常。
-   **完整性原则**：应力求列举全部已知或计划中的接口。对于尚不支持但规划中的能力，可在“规格限制”章节中说明。

### 3. 规格限制
-   **核心内容**：以表格形式集中说明由**实现机制、技术选型或当前阶段决策**所导致的限制、约束或不支持的功能。
-   **列举内容**（包括但不限于）：
    -   性能边界（如QPS、吞吐量上限、响应时间）。
    -   容量限制（如单次处理数据量、支持的最大连接数、最大存储量）。
    -   兼容性限制（如仅支持特定版本的操作系统、浏览器、协议或客户端）。
    -   明确暂不支持的功能特性。
    -   已知的、在可预见未来不会解决的局限性。

### 4. 假设与依赖
-   **核心内容**：说明该功能**正常工作的外部前提条件**或**所依赖的外部服务、组件、数据格式**。这与“规格限制”（自身不支持的能力）形成互补。
-   **描述要求**：以表格形式呈现。
-   **举例**：依赖特定版本的操作系统、需要访问某外部认证服务或计费网关、要求输入数据符合特定的预定义Schema。

### 5. 配置项
-   **核心内容**：说明所有与该功能相关的、可供用户或管理员调整的配置参数。
-   **描述要求**：以表格形式呈现，包含配置路径（如配置文件节点、环境变量名）、配置项名称、可选值/范围、默认值及配置说明。应区分启动时配置与运行时动态配置。

### 6. 实现原理
-   **核心内容**：采用**自顶向下、逐层递进**的树形结构，剖析功能的内部实现逻辑与模块组成。这是理解系统复杂性和设计权衡的关键。
-   **撰写规范**：
    1.  **从外到内**：从**用户/开发者直接交互的实体和接口**开始描述其行为。
    2.  **模块分解**：然后将该实体分解为更内部的**子模块或服务**，描述它们之间的接口与行为。
    3.  **正交性**：对于逻辑上独立、不属于严格上下级关系的**正交子系统**，应单独设立平行章节（如 `6.3 缓存服务` 和 `6.4 消息队列`）进行描述，并阐明其间的协作关系。*（示例：在数据库混合搜索功能中，“查询优化器”、“向量索引引擎”、“全文检索引擎”可视为三个需要单独描述的正交子系统）*
    4.  **标题命名**：章节标题应使用**用户和开发人员容易理解的模块或组件名称**（如“API网关”、“查询优化器”、“存储引擎”），而非“外层实体”、“内部模块”这类抽象概念。

### 7. DFX (Design for X) 考量
-   **核心内容**：对功能所涉及的非功能性需求（设计约束）进行调研与分析，为系统设计提供目标与约束。
-   **调研维度**：需结合功能特点，覆盖但不限于下表所列维度。

| 考量维度 | 调研要点举例 |
| :--- | :--- |
| **性能** | 吞吐量(QPS/TPS)、延迟(P99/P95/平均)、资源消耗(CPU/内存/磁盘IO/网络)。 |
| **可靠性/可用性** | 设计可用性目标(SLA)，容错机制（重试、熔断、降级），故障恢复时间(RTO)，数据恢复点(RPO)。 |
| **可扩展性** | 水平/垂直扩展方案，数据与请求的分布策略（分库分表、分区），扩容是否需停机。 |
| **安全性** | 认证与授权机制，数据加密（传输中/静态），审计日志，防注入攻击等安全策略。 |
| **可维护性** | 监控指标(Metrics)、日志(Logs)、跟踪(Traces)，配置热更新，版本兼容与平滑升级策略。 |
| **成本** | 资源使用成本（如外部API调用费用）、许可费用、存储与计算成本估算模型。 |

### 8. 使用示例
-   **核心内容**：提供简洁、典型的使用示例代码或操作步骤。
-   **撰写要点**：示例应能直接验证主要接口的使用方法，加深读者对功能的理解。可覆盖最常见或最核心的使用场景。

### 9. 参考资料
-   **核心内容**：明确列出本次调研所依据的**所有信息来源**，如产品官方文档、设计草案、会议纪要、RFC、开源代码仓库文件路径等。
-   **重要原则**：**严禁无依据的推测**。调研应优先基于官方文档、源码；其次参考技术文章、社区讨论。如果对某些接口或行为不确定，应在“规格限制”章节中标注为“未明确”或“待确认”，并在此章节说明已查阅但未包含相关信息的来源。

### 10. 评审与确认（建议流程）
-   **核心内容**：本调研报告完成后，建议经项目技术负责人、架构师及相关模块负责人评审，确认无误后作为基线文档归档，并进入后续设计阶段。

| 配置路径 | 配置项 | 可选值/范围 | 默认值 | 说明 |

## 6. 实现原理

### 6.1 [前端模块/服务名称，例如：API网关]
负责接收和处理外部请求。

#### 6.1.1 请求处理流程
1.  接收来自客户端的HTTP/GRPC请求。
2.  对请求进行认证、鉴权与限流。
3.  将请求参数转换为内部调用格式，转发给 **[6.2 后端服务]** 的相应接口。

### 6.2 [后端核心服务，例如：订单服务]
负责核心业务逻辑处理。

#### 6.2.1 订单创建接口
-   **行为**：验证参数，创建订单记录，调用支付服务。
-   **涉及实体**：订单数据模型、库存服务客户端、支付服务客户端。

### 6.3 [正交子系统，例如：缓存服务]
独立负责热点数据的快速存取，提升读性能。

#### 6.3.1 缓存读写接口
-   **行为**：提供 `get(key)`, `set(key, value, ttl)` 等原子操作。
-   **涉及实体**：Redis集群客户端、本地缓存。

### 6.4 [正交子系统，例如：消息队列]
独立负责应用解耦与异步任务处理。

#### 6.4.1 消息生产与消费接口
-   **行为**：订单创建后，向`order.created`主题发布消息；库存服务监听该主题进行扣减。
-   **涉及实体**：Kafka/RabbitMQ生产者与消费者。

## 7. DFX 考量
-   **性能**：订单创建接口P99延迟要求<200ms。缓存命中率需>95%。
-   **可靠性**：核心下单链路可用性99.99%。支付调用失败时，订单状态为“待支付”，并进入重试队列。
-   **可扩展性**：订单表按用户ID哈希分片，支持水平扩容。
-   **安全性**：所有接口需Token认证，敏感操作（如支付）需二次验证。
